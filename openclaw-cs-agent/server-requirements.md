## 二、服务器配置推荐

### 2.1 最低配置 vs 推荐配置

| 资源 | 最低配置 | 推荐配置 | 说明 |
|------|---------|---------|------|
| CPU | 4 核 | 8 核 | BGE-M3 推理 + Neo4j Cypher 查询吃 CPU |
| 内存 | 8 GB | 16 GB | 容器空载 4.3GB + 系统 1.5GB + 并发查询缓冲 |
| 磁盘 | 50 GB SSD | 100 GB SSD | 镜像 20GB + 数据 4GB + 模型 2.2GB + 日志 + 余量 |
| 带宽 | 5 Mbps | 10 Mbps | 首次拉镜像约 20GB，日常 API 调用带宽需求低 |
| 系统 | Ubuntu 22.04 LTS | Ubuntu 22.04 LTS | Docker 兼容性最好 |

### 2.2 为什么最低 8GB 内存

```
容器空载内存      4.3 GB
系统 + Docker     1.5 GB
并发查询缓冲      1.0 GB（BGE-M3 推理 + Neo4j 查询 + LLM 上下文拼接）
安全余量          1.2 GB
─────────────────────────
合计              8.0 GB
```

8GB 能跑，但并发超过 3-5 个请求时可能出现 OOM。16GB 更稳妥。

### 2.3 云服务器价格参考（阿里云/腾讯云，2026 年 4 月）

| 规格 | 月费 | 适用场景 |
|------|------|---------|
| 4C8G（最低可跑） | ¥150-250 | 单客户、低并发（<5 QPS） |
| **4C16G（推荐单客户）** | **¥250-400** | 单客户、中等并发（5-15 QPS） |
| 8C16G | ¥400-600 | 2-3 个客户共享部署 |
| 8C32G | ¥600-1,000 | 5+ 客户，高并发场景 |

> 以上为按量付费参考价。包年包月、竞价实例可再降 30-60%。

## 三、BGE-M3 优化方案（降低服务器成本的关键）

BGE-M3 本地推理是整个系统最重的单一组件：

| 指标 | 占用 |
|------|------|
| 运行内存 | 1.19 GB（占总内存 28%） |
| 镜像大小 | 8.45 GB（占总镜像 46%） |
| 模型文件 | 2.19 GB |
| 首次启动 | 60+ 秒（模型加载） |

### 方案对比

| 方案 | 内存节省 | 磁盘节省 | 月费增加 | 延迟影响 | 检索质量 | 推荐场景 |
|------|---------|---------|---------|---------|---------|---------|
| **A. 硅基 Qwen3-Embedding-0.6B** | -1.2 GB | -10.6 GB | <¥1 | +100-200ms | **更好**（MMTEB +5） | ✅ 默认方案 |
| B. 阿里百炼 text-embedding-v4 | -1.2 GB | -10.6 GB | <¥1 | +100-200ms | 同级 | 备选 fallback |
| C. 保持本地 BGE-M3 | 0 | 0 | ¥0 | 最低 | 基准 | 数据安全要求高 |

### 本地 BGE-M3 实测延迟（i5-12600KF CPU 模式）

| 场景 | 延迟 | 说明 |
|------|------|------|
| 单条短文本（查询） | ~100 ms | 5 次平均，排除首次冷启动 |
| 单条长文本（~500 字） | ~340 ms | 3 次平均 |
| 批量 16 条 | ~544 ms（每条 34ms） | 3 次平均 |

### MMTEB 基准测试对比（来源：Qwen3 Embedding 论文 arxiv:2506.05176）

| 模型 | 参数量 | 维度 | MMTEB 多语言 | MTEB 英文 | CMTEB 中文 | 代码检索 |
|------|--------|------|-------------|----------|-----------|---------|
| BGE-M3 | 0.6B | 1024 | 59.56 | — | — | — |
| **Qwen3-Embedding-0.6B** | 0.6B | 1024 | **64.33** | **70.70** | **66.33** | **75.41** |
| Qwen3-Embedding-4B | 4B | 2560 | 69.45 | 74.60 | 72.26 | 80.06 |
| Qwen3-Embedding-8B | 8B | 4096 | 70.58 | 75.22 | 73.83 | 80.68 |

> Qwen3-Embedding-0.6B 与 BGE-M3 同参数量、同维度，MMTEB 高 5 分，效果更好而非"一致"。

### 方案 A 详情：API Embedding 替代

可选的 Embedding API 服务：

| 服务商 | 模型 | 维度 | 价格（元/百万 token） | MMTEB 分数 | 说明 |
|--------|------|------|---------------------|-----------|------|
| **硅基流动** | Qwen3-Embedding-0.6B | 1024 | ~¥0.07 | **64.33** | ✅ 推荐，效果优于 BGE-M3 |
| 硅基流动 | Qwen3-Embedding-4B | 2560 | ~¥0.14 | 69.45 | 更高质量，需改维度 |
| 硅基流动 | Qwen3-Embedding-8B | 4096 | ~¥0.28 | 70.58 | 最高质量，需改维度 |
| 阿里百炼 | text-embedding-v4 | 1024 | ~¥0.7 | — | 国内直连，兼容 OpenAI API |
| Cloudflare | bge-m3 | 1024 | 免费 | 59.56 | Workers AI 免费额度内 |

**实际月费：** 中度使用（200 次/天）约 ¥0.02/月，重度使用（1,000 次/天）约 ¥0.08/月，几乎可忽略。

**如果采用方案 A：**
- 删除 bge-m3-server 容器
- 修改 knowledge-indexer 和 Dify Chatflow 中的 Embedding 调用地址
- 服务器最低配置降为 **4C8G（¥150/月）**
- 每月 Embedding API 费用约 ¥5-20（取决于知识库更新频率和查询量）

### 方案选择建议

| 客户类型 | 推荐方案 | 服务器配置 | 月费 |
|---------|---------|-----------|------|
| 一般企业客户（默认） | A. 硅基 Qwen3-Embedding-0.6B | 4C8G | ¥150-250 + API <¥1 |
| 数据安全要求高（金融/医疗） | C. 本地 BGE-M3 | 4C16G | ¥250-400 |
| 预算极低的小微企业 | A. 硅基 Qwen3-Embedding-0.6B | 2C4G（极限） | ¥80-120 + API <¥1 |

> 注意：切换 Embedding 模型后必须全量重建 Qdrant 索引（~20 分钟），两个模型的向量空间不兼容。

## 四、Neo4j 优化方案：迁移到 AuraDB Free

Neo4j 是第二大内存消耗方（832 MB），但实际数据量很小。

### 当前 Neo4j 使用情况

| 指标 | 数值 |
|------|------|
| 节点数 | 13,817（Code 12,706 + Concept 1,111） |
| 关系数 | 12,848（CONTAINS 10,902 + CALLS 1,946） |
| 节点标签 | 2 种（Code、Document） |
| 关系类型 | 2 种（CONTAINS、CALLS） |
| 节点属性 | ~17 个字段（name, kind, docstring, signature 等） |
| 查询延迟 | 90-140ms（本地热查询） |
| 内存占用 | 832 MB |
| 磁盘占用 | 533 MB 数据 + 987 MB 镜像 |

### 决策：迁移到 Neo4j AuraDB Free

| 指标 | 本地 Neo4j | AuraDB Free |
|------|-----------|-------------|
| 内存占用 | 832 MB | **0** |
| 磁盘占用 | 1.5 GB | **0** |
| 查询延迟 | 90-140ms | 200-500ms（海外服务器） |
| 对总链路影响 | — | +100-400ms（30 秒链路中 <2%） |
| 免费额度 | — | 200K 节点 / 400K 关系（当前仅用 7%） |
| Cypher 兼容性 | — | 100%，只改连接地址 |
| 月费 | 含在服务器里 | **¥0** |

### 迁移步骤

1. 注册 Neo4j AuraDB Free（https://neo4j.com/cloud/aura-free/）
2. 创建免费实例，获取连接 URI + 用户名 + 密码
3. 运行 restore_neo4j.py 将 graph.json 数据导入 AuraDB（改连接参数）
4. 修改 Dify Chatflow 代码节点中的 Neo4j 连接地址
5. 修改 docker-compose.yml 注释掉 neo4j 服务
6. 验证查询正常

### 注意事项

- AuraDB Free 无 SLA，不适合对可用性要求极高的生产环境
- 免费版服务器在海外，国内延迟 200-500ms，在 30 秒总链路中可接受
- 如需更低延迟或 SLA 保障，可升级 AuraDB Professional（$65/月起）
- 付费客户如对延迟敏感，可保留本地 Neo4j 方案作为选项

## 八、运行时资源监控建议

部署后建议监控以下指标：

| 指标 | 告警阈值 | 说明 |
|------|---------|------|
| 内存使用率 | > 85% | 接近 OOM，需要扩容或优化 |
| CPU 使用率 | > 80%（持续 5 分钟） | BGE-M3 推理或 Neo4j 查询瓶颈 |
| 磁盘使用率 | > 80% | MySQL binlog + Neo4j 日志增长 |
| BGE-M3 响应时间 | > 2 秒 | 模型推理变慢，可能内存不足 swap |
| Neo4j 查询时间 | > 3 秒 | 图谱过大或查询未优化 |
| Dify API 响应时间 | > 30 秒 | 整体链路超时 |

### 简易监控命令

```bash
# 实时查看容器资源
docker stats

# 查看特定容器日志
docker logs -f --tail 100 hmsdkbot-bge-m3
docker logs -f --tail 100 hmsdkbot-neo4j

# 检查磁盘
df -h
du -sh /path/to/data/*
```
