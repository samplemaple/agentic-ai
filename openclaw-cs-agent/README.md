## 二、从"内部工具"到"可交付产品"的差距

hmsdkbot 目前是为你自己公司的 SDK 文档定制的，要变成可卖给其他企业的产品，需要解决：

### 2.1 知识库通用化（核心）

**现状：** 知识库内容是硬编码的 HMViewer SDK 文档，构建流程依赖特定的代码解析器（Java/Kotlin/C）。

**目标：** 客户上传任意文档（PDF/Word/Markdown），系统自动构建知识库。

| 改造项 | 现状 | 目标 |
|--------|------|------|
| 文档输入 | 手动转换 docx → 21 个 md 文件 | 客户上传 PDF/Word/MD，自动解析 |
| 知识图谱 | code-parser 解析 Java/Kotlin 源码 | 通用文档 → LLM 语义提取 → 图谱 |
| 向量索引 | 手动运行 build_qdrant_index.py | 上传后自动分块 + 索引 |
| 知识库管理 | 命令行脚本 | admin-dashboard 中可视化管理 |

### 2.2 多租户支持

**现状：** 单租户，所有数据混在一起。

**目标：** 每个客户独立的知识库、对话历史、配置。

| 改造项 | 方案 |
|--------|------|
| 知识库隔离 | Qdrant Collection 按客户 ID 分离 |
| Neo4j 隔离 | 节点增加 tenant_id 属性，查询时过滤 |
| 对话隔离 | Dify 已支持 conversation_id 隔离 |
| 配置隔离 | MySQL 增加 tenant 表，关联 API Key |

### 2.3 部署简化

**现状：** 14 个容器，手动配置 .env，手动导入 Dify Chatflow。

**目标：** 一键部署，客户零配置。

| 改造项 | 方案 |
|--------|------|
| 初始化 | setup.sh 一键完成所有配置 |
| Dify Chatflow | 自动通过 API 导入 DSL |
| 模型配置 | 客户只需提供 DeepSeek API Key |
| 健康检查 | 启动后自动验证所有服务 |

### 2.4 企微接入简化

**现状：** wecom-adapter 已完成，但需要客户配置企微凭证 + 公网回调。

**目标：** 提供配置向导，降低接入门槛。

## 七、关键决策点

### Q1：是否需要从 Dify 迁移？

**建议：暂时不迁移。** Dify 的 Chatflow 可视化编排是你的卖点之一（客户可以看到工作流，增加信任感）。等客户量上来后再考虑自研替代。

### Q2：是否需要支持微信（非企微）？

**建议：先只做企微。** 企微有官方 API，合规无风险。微信个人号接入灰色地带，不适合商业交付。

### Q3：服务器部署在哪？

**建议：**
- Demo 阶段：你的本地电脑（已有 Docker 环境）
- 交付客户：阿里云 4C8G ECS（¥150-250/月），优化后最低 2C4G（¥80-120/月）
- 多客户共享：4C8G 可支撑 2-3 个客户

> 优化后（BGE-M3 → API + Neo4j → AuraDB Free）：17 容器降到 15 个，内存从 4.3GB 降到 2.3GB
> 详细配置分析见 [server-requirements.md](./server-requirements.md)

### Q4：BGE-M3 本地推理还是换 API？

**决策：默认用硅基流动 Qwen3-Embedding-0.6B API。**

| 维度 | 本地 BGE-M3 | 硅基 Qwen3-Embedding-0.6B |
|------|-----------|-------------------------|
| MMTEB 多语言得分 | 59.56 | **64.33**（+5 分） |
| 参数量 / 维度 | 0.6B / 1024 | 0.6B / 1024（完全兼容） |
| 单条查询延迟 | ~100ms | ~200-300ms |
| 对总链路影响 | — | +100-200ms（30 秒链路中 <1%） |
| 运行内存 | 1.2 GB | 0 |
| 磁盘（镜像+模型） | 10.6 GB | 0 |
| 月费 | ¥0 | <¥1 |
| 服务器可降级 | 4C16G | **4C8G（省 ¥100-150/月）** |

数据安全要求高的客户（金融/医疗）可选本地部署方案，需要 4C16G。

> 详细对比见 [server-requirements.md](./server-requirements.md)
