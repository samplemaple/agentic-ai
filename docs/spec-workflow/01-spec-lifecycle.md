# Spec 完整生命周期（五阶段）

> 每个 Spec 从规划到归档的标准流程。
> 5 个阶段，每个阶段有明确的输入、输出和检查点。

---

## 流程总览

```
阶段1          阶段2          阶段3          阶段4          阶段5
准备+澄清  →  三件套生成  →  质量保障  →  执行任务  →  验证+归档
    ↓             ↓             ↓             ↓             ↓
提问确认      需求设计任务    检查清单      按任务块      测试文档
```

**Session 分配建议**：
- **设计 session**（阶段 1-3）：准备→澄清→三件套→质量保障，保持上下文连贯
- **执行 session**（阶段 4）：按任务块分 session，每个任务块独立执行
- **验证 session**（阶段 5）：用户验证 + AI 归档

---

## 阶段 1：准备 + 澄清

### 目标
确保 AI 完全理解需求，再开始生成文档。

### 输入
- 用户的功能描述/提示词
- 项目现有文档（如有）

### 动作

#### 1.1 准备（AI 自动）
1. 读取项目相关文档，了解上下文
2. 识别该 spec 涉及的技术领域
3. 检查是否有可复用的现有系统

#### 1.2 澄清提问（AI 必须做）
AI 必须先生成 5-8 个澄清问题，用户回答后才能进入下一阶段。

**标准问题清单**（根据 spec 复杂度调整）：

1. **核心场景**：这个功能的主要使用场景是什么？
2. **边界条件**：哪些情况应该处理？哪些不应该处理？
3. **优先级**：如果有时间压力，哪些可以砍掉？
4. **依赖**：这个功能依赖哪些现有系统？
5. **验收标准**：怎样算"完成"？
6. **技术约束**：有没有必须遵循的技术限制？
7. **扩展性**：未来可能如何扩展？
8. **风险**：你觉得哪些地方可能有坑？

**可选问题**（根据 spec 类型）：
- 涉及 UI：交互风格、视觉偏好？
- 涉及性能：性能目标、瓶颈在哪？
- 涉及安全：安全要求、威胁模型？
- 涉及集成：第三方 API、数据格式？

#### 1.3 复杂度评估（AI 自动）
基于用户回答，AI 评估并告知：

```markdown
## 复杂度评估

基于你的回答，我评估这个 spec：
- 任务数量：约 {N} 个
- 预估上下文：~{X}k token
- 风险等级：{低/中/高}
- 涉及现有文件：{N} 个

建议：{不分批 / 分 2-3 个任务块}
```

### 输出
- 澄清问题清单 + 用户回答
- 复杂度评估结果
- 对 spec 范围和约束的完整理解

### 检查点
- [ ] AI 是否问了 5-8 个澄清问题？
- [ ] 用户是否回答了所有问题？
- [ ] 复杂度评估是否合理？

## 阶段 2：三件套生成

### 目标
生成完整的需求、设计、任务文档。

### 输入
- 阶段 1 的澄清结果
- 项目现有文档

### 动作

#### 2.1 生成 requirements.md
基于用户回答，生成需求文档。

**必须包含**：
- 简介（一句话定位）
- 术语表（新增概念）
- 需求列表（用户故事 + 验收标准）
- 边界（明确不做的内容）

**格式要求**：
- 每个需求有用户故事："作为{角色}，我想要{功能}，以便{价值}"
- 每个需求有可验证的验收标准："给定{前提}，当{操作}，则{结果}"
- 术语表覆盖所有新增概念

#### 2.2 生成 design.md
基于 requirements.md，生成设计文档。

**必须包含**：
- 架构图（组件关系）
- 组件接口（公共 API）
- 数据模型（数据结构）
- 正确性属性（核心逻辑的不变量）
- 错误处理策略
- 测试策略（单元/集成/端到端）

**推荐包含**：
- 竞品分析（3-5 款同类产品的做法）
- 扩展性设计（未来如何扩展）
- 性能考虑（如有性能要求）

#### 2.3 生成 tasks.md
基于 design.md，生成任务列表和执行计划。

**必须包含**：
- 执行计划（任务块划分）
- 任务详情（按依赖顺序）

**高风险改造独立成批**：
以下改造必须独立成一个任务块，不能与其他任务混合：

| 高风险类型 | 说明 | 示例 |
|-----------|------|------|
| 构造函数签名变更 | 影响所有实例化代码 | 新增必填参数、删除参数、修改参数类型 |
| 继承链修改 | 影响多态调用 | 修改基类、改变继承关系、新增抽象方法 |
| 接口签名变更 | 影响所有实现类 | 新增接口方法、修改方法签名、删除接口 |
| 枚举值变更 | 影响所有 switch/case | 插入枚举值（会打乱索引）、删除枚举值 |
| 数据库 schema 变更 | 影响数据迁移 | 新增/删除列、修改列类型、新增表 |
| 核心组件重构 | 影响所有依赖 | 重命名核心类、移动命名空间、拆分模块 |
| API 接口变更 | 影响前后端契约 | 修改请求/响应格式、删除接口、修改 URL |
| 依赖注入变更 | 影响所有服务 | 修改容器配置、更换 DI 框架 |

**执行计划格式**：

```markdown
## 执行计划

| 任务块 | 包含任务 | 预估上下文 | 依赖 | 风险 | 状态 |
|--------|---------|-----------|------|------|------|
| A | task1, task2, task3 | ~100k | 无 | 低 | ⬜ |
| B | task4（接口重构） | ~80k | A完成 | **高** | ⬜ |
| C | task5, task6 | ~120k | B完成 | 低 | ⬜ |
| D | task7（测试） | ~50k | C完成 | 低 | ⬜ |

### 任务块 A：{标题}（低风险）
- 产出文件：{文件列表}
- 修改文件：{文件列表}
- 验证：{编译/测试命令}

### 任务块 B：{标题}（高风险）
- **风险类型**：{构造函数变更/继承链修改/接口签名变更/...}
- **影响范围**：{受影响的文件/模块}
- **验证方法**：{如何验证改造正确性}
- **回滚方案**：{出错如何恢复}
- 产出文件：{文件列表}
- 修改文件：{文件列表}
- 验证：{编译/测试命令}

### 任务块 C：{标题}（低风险）
...
```

**任务详情格式**：

```markdown
## 任务详情

### task1: {标题}
**产出**：{文件路径} — {一句话描述}
**依赖**：无
**验证**：{编译/测试命令}

### task2: {标题}
**产出**：{文件路径} — {一句话描述}
**依赖**：task1
**验证**：{编译/测试命令}
```

### 输出
- `requirements.md` — 需求文档
- `design.md` — 设计文档
- `tasks.md` — 任务列表 + 执行计划

### 检查点
- [ ] requirements.md 是否覆盖了用户回答中的所有要点？
- [ ] design.md 是否有架构图和接口定义？
- [ ] tasks.md 是否有执行计划和任务块划分？
- [ ] 任务块的预估上下文是否合理（100k-150k）？
- [ ] 高风险改造是否独立成批？
- [ ] 任务依赖关系是否正确（无循环）？
- [ ] 用户是否确认了三件套？

## 阶段 3：质量保障

### 目标
检查三件套质量，生成实施注意事项。

### 输入
- 完成的三件套
- 用户确认

### 动作

#### 3.1 生成 implementation-notes.md
从三件套中提炼执行阶段需要的关键信息。

**标准章节**（按需选用）：
1. 关键设计决策及理由
2. 现有代码改造清单
3. 高风险改造说明
4. 跨模块联动约束
5. 测试注意事项
6. 文件组织（新增 + 修改）

#### 3.2 检查清单
AI 必须逐条检查：

**需求检查**：
- [ ] 每个需求有用户故事和验收标准
- [ ] 术语表覆盖所有新增概念
- [ ] 边界明确（哪些不做）

**设计检查**：
- [ ] 架构图清晰展示组件关系
- [ ] 接口定义完整（参数、返回值、异常）
- [ ] 数据模型覆盖所有场景
- [ ] 错误处理策略明确

**任务检查**：
- [ ] 任务按依赖顺序排列
- [ ] 每个任务有明确的产出和验证方式
- [ ] 任务块划分合理（上下文预算）
- [ ] 高风险任务独立成块

**implementation-notes 检查**：
- [ ] 现有代码改造清单是否完整（用搜索验证）
- [ ] 高风险改造是否有验证方案
- [ ] 预留功能是否说明了 fallback 行为

#### 3.3 现有代码审查
如果 spec 涉及修改现有代码：

1. 搜索所有被修改的方法/类的调用点
2. 判断每个调用点是否需要改造
3. 列入 implementation-notes.md 的改造清单

### 输出
- `implementation-notes.md` — 实施注意事项
- 检查清单（全部通过）

### 检查点
- [ ] implementation-notes.md 是否覆盖了所有关键决策？
- [ ] 现有代码改造清单是否完整？
- [ ] 检查清单是否全部通过？
- [ ] tasks.md 顶部是否有引用 implementation-notes.md？

---

## 阶段 4：执行任务

### 目标
按任务块分 session 执行，每个任务块完成后验证。

### 输入
- tasks.md（含执行计划）
- implementation-notes.md
- requirements.md + design.md（参考）

### 动作

#### 4.1 读取执行计划
1. 读 tasks.md 的执行计划
2. 找到当前待执行的任务块
3. 读取该任务块的详细说明

#### 4.2 执行任务块
按任务块中的任务顺序执行：

1. 读取需要修改的现有文件
2. 实现产出文件
3. 编写测试文件
4. 运行验证命令（编译 + 测试）
5. 如果失败，修复后重新验证

#### 4.3 更新进度
任务块完成后：

1. 在 tasks.md 中更新任务块状态：`⬜` → `✅`
2. 追加"已完成任务块摘要"：

```markdown
### 任务块 A 已完成（{日期}）

**产出文件**：
- {文件1} — {描述}
- {文件2} — {描述}

**修改文件**：
- {文件1} — {改动摘要}
- {文件2} — {改动摘要}

**验证结果**：
- 编译：✅ 通过
- 测试：✅ 通过（{N} 个用例）

**关键决策**：
- {决策1} — {理由}
- {决策2} — {理由}

**踩坑记录**：
- {问题} — {解决方案}
```

#### 4.4 启动下一个任务块
如果还有未完成的任务块：

1. 开新 session
2. 复制以下启动指令：

```
执行 {spec-name} 的下一个待执行任务块。

步骤：
1. 读 tasks.md 执行计划，确认当前待执行任务块
2. 读该任务块的详细说明（产出、修改、验证）
3. 读需要修改的现有文件
4. 实现产出 + 测试
5. 运行验证命令
6. 更新 tasks.md：任务块状态改为 ✅、追加已完成摘要
```

### 输出
- 所有代码文件（新增 + 修改）
- 所有测试文件
- 更新后的 tasks.md

### 检查点
- [ ] 每个任务块完成后编译通过？
- [ ] 每个任务块完成后测试通过？
- [ ] 已有测试无回归失败？
- [ ] tasks.md 状态已更新？

---

## 阶段 5：验证 + 归档

### 目标
用户验证功能，AI 归档文档。

### 输入
- 阶段 4 产出的代码
- 开发过程中的经验和踩坑

### 动作

#### 5.1 用户验证
用户手动验证：

1. 运行完整测试套件
2. 手动测试核心功能
3. 检查边界情况
4. 记录发现的问题

**如果发现问题**：
- 记录 bug 清单（现象、复现步骤、预期行为）
- 回到阶段 4 修复
- 修复后重新验证

#### 5.2 Bug 处理（如有）
按优先级处理：

| 类型 | 优先级 | 处理方式 |
|------|--------|---------|
| 编译错误 | P0 立即修 | 同 session 修复 |
| 测试失败 | P0 立即修 | 同 session 修复 |
| 功能不正确 | P1 本轮修 | 同 session 或新 session |
| 视觉/体验问题 | P2 可延后 | 记录到待办 |
| 性能问题 | P3 后续优化 | 记录到待办 |

**经验沉淀**：
每个 bug 修完后判断是否需要沉淀到项目规范：
- 通用性踩坑 → 追加到代码规范
- 测试盲区 → 补充测试用例
- 文档不准确 → 修正 implementation-notes.md

#### 5.3 归档
验证通过后，AI 执行归档：

1. **更新项目进度文档**
   - 标记 spec 为已完成
   - 更新下一步计划

2. **更新项目规范**（如有新规则）
   - 代码规范追加踩坑记录
   - 预防性规则追加新条目

3. **精简 spec 文档**
   - 将详细说明替换为摘要
   - 详细内容迁移到归档目录

4. **更新文件索引**
   - 新增文件加入项目结构文档
   - 更新模块关联关系

### 输出
- 验证通过的功能
- 更新后的项目文档
- 沉淀的经验教训

### 检查点
- [ ] 所有 P0/P1 bug 已修复？
- [ ] 测试全部通过？
- [ ] 用户验证通过？
- [ ] 项目文档已更新？
- [ ] 经验已沉淀？

## 流程速查表

| 阶段 | 核心动作 | 必须做 | 可选做 |
|------|---------|--------|--------|
| 1 准备+澄清 | 提问 5-8 个问题 | ✅ 澄清问题 | 复杂度评估 |
| 2 三件套生成 | 生成需求+设计+任务 | ✅ 三件套 | 竞品调研、扩展性测试 |
| 3 质量保障 | 检查+生成注意事项 | ✅ 检查清单 | 现有代码审查 |
| 4 执行任务 | 按任务块执行 | ✅ 验证通过 | 分批执行 |
| 5 验证+归档 | 用户验证+文档归档 | ✅ 用户验证 | Bug 经验沉淀 |

---

## 与原版 Kiro 的关键区别

| 方面 | 原版 Kiro | 本流程 |
|------|----------|--------|
| 提问 | 隐式 | **显式，必须先问 5-8 个问题** |
| 执行计划 | 无 | **集成在 tasks.md 中** |
| 质量保障 | 无 | **有 implementation-notes + 检查清单** |
| 分批执行 | 无 | **AI 自动评估，按任务块分 session** |
| 归档 | 无 | **有标准化归档流程** |
| 流程阶段 | 3 阶段 | **5 阶段（含准备+质量+归档）** |
