# Spec 提示词模板

> 用于启动新 spec 的提示词模板。
> 使用 `/spec` 命令触发。

---

## 模板

```
/spec {中文名称}
{一句话定位}

职责：
- {职责1}
- {职责2}
- {职责3}

可复用系统：
- {系统1}：{来源}
- {系统2}：{来源}
```

---

## 示例

### 示例 1：用户认证系统

```
/spec 用户认证系统
支持注册、登录、密码重置

职责：
- 用户注册（邮箱+密码）
- 用户登录（JWT token）
- 密码重置（邮件验证）
- 用户信息管理

可复用系统：
- 数据库连接模块：已实现 PostgreSQL 连接池
- 邮件发送模块：已实现 SMTP 发送
- 中间件框架：已实现 Express 中间件
```

### 示例 2：购物车系统

```
/spec 购物车系统
支持商品添加、数量修改、结算

职责：
- 添加商品到购物车
- 修改商品数量
- 删除商品
- 计算总价
- 结算下单

可复用系统：
- 商品模块：已实现商品查询 API
- 库存模块：已实现库存检查 API
- 用户模块：已实现用户认证
```

### 示例 3：通知系统

```
/spec 通知系统
支持多渠道通知发送

职责：
- 通知模板管理
- 多渠道发送（邮件、短信、推送）
- 发送记录和状态追踪
- 通知偏好设置

可复用系统：
- 邮件模块：已实现 SMTP 发送
- 用户模块：已实现用户认证
- 消息队列：已实现 RabbitMQ 连接
```

---

## 填写说明

### 中文名称
- 简洁明了，5-10 个字
- 体现模块核心功能
- 示例：用户认证、购物车、通知系统

### 一句话定位
- 一句话描述模块的核心价值
- 示例："支持用户注册、登录、密码重置的认证模块"

### 职责
- 列出 3-5 个核心职责
- 每个职责一句话
- 避免过于详细（详细内容在澄清阶段讨论）

### 可复用系统
- 列出所有可复用的现有系统
- 说明来源（哪个模块/哪个 spec）
- AI 会自动判断如何集成

---

## 注意事项

1. **保持简洁**：提示词控制在 15 行以内
2. **列出可复用系统**：这是最重要的部分，避免 AI 重新造轮子
3. **不要过于详细**：提示词只是启动 spec 的入口，详细内容在澄清阶段讨论
4. **使用 /spec 命令**：这是标准的触发方式，AI 会自动进入提问模式
