## 示例

```markdown
# 实施注意事项：用户认证系统（user-auth）

> 生成日期：2024-01-15
> 基于三件套

---

## 一、关键设计决策及理由

### 1.1 使用 bcrypt 哈希密码

- **做什么**：使用 bcrypt 算法哈希密码，cost factor 为 12
- **为什么**：bcrypt 是目前最安全的密码哈希算法之一，抗彩虹表攻击
- **不这样做**：使用 MD5 或 SHA256 容易被暴力破解

### 1.2 使用双 token 机制

- **做什么**：Access Token（15分钟）+ Refresh Token（7天）
- **为什么**：平衡安全性和用户体验
- **不这样做**：
  - 只用 Access Token：过期后用户需要重新登录
  - 只用 Refresh Token：安全风险高

### 1.3 邮箱验证可选

- **做什么**：注册后可以立即登录，邮箱验证可选
- **为什么**：降低注册门槛，提高转化率
- **不这样做**：强制邮箱验证会流失大量用户

---

## 二、现有代码改造清单

| 文件 | 当前代码 | 改造后代码 | 改动原因 |
|------|---------|-----------|---------|
| `src/routes/index.ts` | 只有健康检查路由 | 添加 `/api/auth/*` 路由 | 挂载认证接口 |
| `src/middleware/error-handler.ts` | 通用错误处理 | 添加认证错误处理 | 处理认证相关错误 |

---

## 三、高风险改造

### 3.1 密码哈希性能

- **风险**：bcrypt 计算耗时高，高并发时可能导致请求超时
- **验证方法**：压力测试，监控响应时间
- **回滚方案**：降低 cost factor 到 10

### 3.2 JWT Secret 管理

- **风险**：Secret 泄露导致 token 可伪造
- **验证方法**：检查 Secret 是否硬编码
- **回滚方案**：立即更换 Secret，使所有 token 失效

---

## 四、跨模块联动约束

### 本模块为其他模块预留

- `verifyToken` 接口 — 供其他模块验证用户身份
- `getCurrentUser` 接口 — 供其他模块获取当前用户

### 其他模块给本模块的待办

- 无

### 预留但不实现的功能

- 第三方登录 — fallback：提示"暂不支持"
- 手机号登录 — fallback：提示"暂不支持"
- MFA — fallback：提示"暂不支持"

---

## 五、测试注意事项

### 单元测试

- 密码哈希测试需要 mock bcrypt，否则测试耗时长
- JWT token 测试需要固定时间，否则 token 过期时间不确定

### 集成测试

- 数据库测试需要使用测试数据库，不要污染开发数据库
- 每个测试后清理测试数据

### 端到端测试

- 使用 supertest 测试 API，不需要启动真实服务器
- 测试并发场景时，需要等待所有请求完成
```

## 填写说明

### 关键设计决策及理由

**写法要点**：
- 每个决策一小节
- 说明"做什么 + 为什么 + 不这样做会怎样"
- 重点标注来自踩坑记录的决策

**常见决策类型**：
- 技术选型（为什么用 bcrypt 而不是 MD5）
- 架构决策（为什么用双 token 而不是单 token）
- 业务决策（为什么邮箱验证可选）

### 现有代码改造清单

**写法要点**：
- 这是最易出错的部分，必须完整
- 用搜索验证所有调用点
- 新增文件不会错，改错现有文件会导致回归 bug

**搜索方法**：
- 搜索被修改的方法名
- 搜索被修改的类名
- 搜索被修改的接口名

### 高风险改造

**写法要点**：
- 涉及核心组件改造时必须有
- 详细说明兼容性风险
- 提供验证方法和回滚方案

**常见高风险场景**：
- 性能敏感操作（密码哈希）
- 安全敏感操作（JWT Secret）
- 数据库 schema 变更
- 接口签名变更

### 跨模块联动约束

**写法要点**：
- 明确标注"本模块做"和"其他模块做"的边界
- 预留功能必须说明 fallback 行为

**常见联动场景**：
- 本模块为其他模块提供接口
- 其他模块依赖本模块的数据
- 预留但不实现的功能

### 测试注意事项

**写法要点**：
- Mock 策略
- 测试数据准备
- 测试环境配置
- 已知的测试限制

**常见注意事项**：
- 需要 mock 外部依赖（数据库、第三方 API）
- 需要准备测试数据
- 需要清理测试环境
- 某些场景难以测试（并发、时间相关）

### 文件组织

**写法要点**：
- 列出所有新增文件
- 列出所有修改文件
- 简要说明改动内容

**格式要求**：
- 使用表格
- 文件路径要准确
- 说明要简洁

---

## 注意事项

1. **决策要有理由**：不能只写"用 bcrypt"，要写"为什么用 bcrypt"
2. **改造清单要完整**：用搜索验证，不要遗漏
3. **高风险要重视**：提供验证方法和回滚方案
4. **联动要清晰**：明确边界，避免职责不清
5. **测试要实用**：提供具体的注意事项，不是泛泛而谈
6. **文件组织要准确**：路径要准确，说明要简洁
