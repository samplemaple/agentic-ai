# 任务列表模板

> 用于生成 tasks.md 的模板。
> AI 根据 design.md，使用此模板生成任务列表和执行计划。
> 此模板将执行计划（原 progress.md）整合到 tasks.md 中。

---

## 示例

```markdown
# 任务列表：用户认证系统（user-auth）

> 生成日期：2024-01-15
> 基于 design.md

---

## 执行计划

### 复杂度评估

| 维度 | 评估 |
|------|------|
| 任务数量 | 8 个 |
| 预估上下文 | ~120k token |
| 涉及现有文件 | 2 个 |
| 风险等级 | 中等 |

### 分批策略

分 4 个任务块：
- 任务块 A：数据层（低风险）
- 任务块 B：数据库迁移（高风险 - schema 变更）
- 任务块 C：服务层+API层（低风险）
- 任务块 D：测试（低风险）

### 任务块总览

| 任务块 | 包含任务 | 预估上下文 | 依赖 | 风险 | 状态 |
|--------|---------|-----------|------|------|------|
| A | task1, task2 | ~80k | 无 | 低 | ⬜ |
| B | task3（数据库迁移） | ~40k | A完成 | **高** | ⬜ |
| C | task4, task5, task6 | ~150k | B完成 | 低 | ⬜ |
| D | task7, task8 | ~60k | C完成 | 低 | ⬜ |

### 任务块详情

#### 任务块 A：数据层（低风险）

**产出文件**：
- `src/models/user.ts` — User 数据模型
- `src/repositories/user-repository.ts` — 用户数据访问层
- `src/migrations/001-create-users.ts` — 数据库迁移

**修改文件**：
- `src/config/database.ts` — 添加用户表连接

**测试任务**：
- 测试文件：`tests/repositories/user-repository.test.ts`
  - 创建用户
  - 根据邮箱查找用户
  - 更新用户信息

**验证**：`npm test`

#### 任务块 B：服务层+API层

**产出文件**：
- `src/services/auth-service.ts` — 认证服务
- `src/controllers/auth-controller.ts` — 认证控制器
- `src/middleware/auth-middleware.ts` — 认证中间件

**修改文件**：
- `src/routes/index.ts` — 添加认证路由

**测试任务**：
- 测试文件：`tests/services/auth-service.test.ts`
  - 用户注册
  - 用户登录
  - 密码重置
  - Token 验证

**验证**：`npm test`

#### 任务块 C：测试

**产出文件**：
- `tests/integration/auth.test.ts` — 集成测试
- `tests/e2e/auth.e2e.test.ts` — 端到端测试

**修改文件**：无

**测试任务**：
- 测试文件：`tests/integration/auth.test.ts`
  - 注册→验证→登录 完整流程
  - 密码重置完整流程

**验证**：`npm test`

## 任务详情

### task1: 创建 User 数据模型

**产出**：`src/models/user.ts` — User 数据模型定义
**依赖**：无
**验证**：`npm run build`

**实现要点**：
- 定义 User 接口
- 定义 UserSchema（使用 TypeORM 或 Sequelize）
- 导出模型类

### task2: 创建 UserRepository

**产出**：`src/repositories/user-repository.ts` — 用户数据访问层
**依赖**：task1
**验证**：`npm run build`

**实现要点**：
- 实现 CRUD 方法
- 实现 findByEmail 方法
- 处理数据库错误

### task3: 创建数据库迁移

**产出**：`src/migrations/001-create-users.ts` — 创建用户表
**依赖**：task1
**验证**：`npm run migration:run`

**实现要点**：
- 创建 users 表
- 添加索引
- 设置约束

### task4: 创建 AuthService

**产出**：`src/services/auth-service.ts` — 认证服务
**依赖**：task2
**验证**：`npm run build`

**实现要点**：
- 实现 register 方法
- 实现 login 方法
- 实现 resetPassword 方法
- 实现 verifyToken 方法
- 使用 bcrypt 哈希密码
- 使用 JWT 生成 token

### task5: 创建 AuthController

**产出**：`src/controllers/auth-controller.ts` — 认证控制器
**依赖**：task4
**验证**：`npm run build`

**实现要点**：
- 实现 register 接口
- 实现 login 接口
- 实现 resetPassword 接口
- 参数验证
- 错误处理

### task6: 创建 AuthMiddleware

**产出**：`src/middleware/auth-middleware.ts` — 认证中间件
**依赖**：task4
**验证**：`npm run build`

**实现要点**：
- 验证 JWT token
- 提取用户信息
- 处理 token 过期

### task7: 编写集成测试

**产出**：`tests/integration/auth.test.ts` — 集成测试
**依赖**：task5, task6
**验证**：`npm test`

**实现要点**：
- 测试完整注册流程
- 测试完整登录流程
- 测试密码重置流程
- 测试错误处理

### task8: 编写端到端测试

**产出**：`tests/e2e/auth.e2e.test.ts` — 端到端测试
**依赖**：task7
**验证**：`npm test`

**实现要点**：
- 使用 supertest 测试 API
- 测试正常流程
- 测试异常流程
- 测试并发场景
```

## 填写说明

### 执行计划

#### 复杂度评估
- **任务数量**：统计所有任务
- **预估上下文**：估算所有任务的 token 消耗
- **涉及现有文件**：统计需要修改的现有文件
- **风险等级**：评估技术风险

#### 分批策略
根据复杂度评估决定：
- 任务 < 10，上下文 < 150k → 不分批
- 任务 10-20，上下文 150k-300k → 可选分批
- 任务 > 20，上下文 > 300k → 必须分批

**高风险改造必须独立成批**（无论任务数量）：
- 构造函数签名变更
- 继承链修改
- 接口签名变更
- 数据库 schema 变更
- 核心组件重构

#### 任务块划分原则
- **依赖链顺序**：按依赖关系划分
- **风险隔离**：高风险任务独立成块
- **文件聚合**：修改同一组文件的任务合并
- **测试收尾**：测试任务放最后

### 任务详情

#### 产出
- 明确的文件路径
- 一句话描述文件内容

#### 依赖
- 前置任务编号
- 无依赖写"无"

#### 验证
- 具体的编译/测试命令
- 不能写"运行测试"，要写具体命令

#### 实现要点
- 关键实现步骤
- 注意事项
- 参考代码

### 已完成任务块摘要

执行后填写，包括：
- 产出文件列表
- 修改文件列表
- 验证结果
- 关键决策
- 踩坑记录

---

## 注意事项

1. **执行计划要合理**：任务块划分要符合上下文预算
2. **依赖关系要正确**：不能有循环依赖
3. **任务粒度要适中**：太大拆不开，太小太琐碎
4. **验证命令要具体**：不能写"运行测试"，要写具体命令
5. **实现要点要实用**：关键步骤和注意事项，不是泛泛而谈
6. **状态要及时更新**：任务块完成后立即更新状态
7. **摘要要详细**：便于后续任务块理解前面做了什么
