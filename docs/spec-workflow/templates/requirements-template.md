> 用于生成 requirements.md 的模板。

## 示例

```markdown
# 需求文档：用户认证系统（user-auth）

> 生成日期：2024-01-15
> 基于澄清问题回答

---

## 简介

支持用户注册、登录、密码重置的认证模块，为 Web 应用提供安全的用户身份验证。

## 术语表

| 术语 | 定义 |
|------|------|
| JWT | JSON Web Token，用于无状态认证的令牌格式 |
| Refresh Token | 用于刷新 Access Token 的长期令牌 |
| Hash | 密码哈希，使用 bcrypt 算法加密存储 |

## 需求列表

### 需求 1：用户注册

**用户故事**：
作为新用户，我想要通过邮箱注册账号，以便使用系统功能。

**验收标准**：
- 给定 有效邮箱和密码，当 提交注册表单，then 创建用户账号并发送验证邮件
- 给定 已注册邮箱，当 提交注册表单，then 提示"邮箱已注册"
- 给定 弱密码（<8位），当 提交注册表单，then 提示"密码强度不足"

**优先级**：P0

### 需求 2：用户登录

**用户故事**：
作为已注册用户，我想要通过邮箱和密码登录，以便访问个人数据。

**验收标准**：
- 给定 正确邮箱和密码，当 提交登录表单，then 返回 JWT token 并跳转首页
- 给定 错误密码，当 提交登录表单，then 提示"密码错误"
- 给定 未注册邮箱，当 提交登录表单，then 提示"用户不存在"

**优先级**：P0

### 需求 3：密码重置

**用户故事**：
作为忘记密码的用户，我想要通过邮箱重置密码，以便重新访问账号。

**验收标准**：
- 给定 已注册邮箱，当 请求密码重置，then 发送重置链接邮件
- 给定 未注册邮箱，当 请求密码重置，then 提示"邮箱未注册"（安全考虑，可选）
- 给定 有效重置链接，当 设置新密码，then 更新密码并使旧 token 失效

**优先级**：P1

## 边界

### 本 Spec 做
- 邮箱+密码注册
- 邮箱+密码登录
- 密码重置（邮件验证）
- JWT token 管理

### 本 Spec 不做
- 第三方登录（Google、GitHub）— 留给 Spec 2
- 手机号登录 — 留给 Spec 3
- 多因素认证（MFA）— 留给 Spec 4
- 用户权限管理 — 留给 Spec 5

## 依赖

| 依赖 | 状态 | 说明 |
|------|------|------|
| 数据库模块 | ✅ 已完成 | PostgreSQL 连接池 |
| 邮件模块 | ✅ 已完成 | SMTP 发送 |
| 中间件框架 | ✅ 已完成 | Express 中间件 |

## 验收场景

### 场景 1：新用户注册
1. 打开注册页面
2. 输入邮箱 "test@example.com" 和密码 "password123"
3. 点击"注册"按钮
4. 看到"注册成功，请验证邮箱"提示
5. 收到验证邮件

### 场景 2：用户登录
1. 打开登录页面
2. 输入邮箱 "test@example.com" 和密码 "password123"
3. 点击"登录"按钮
4. 跳转到首页，看到用户信息

### 场景 3：密码重置
1. 打开登录页面
2. 点击"忘记密码"
3. 输入邮箱 "test@example.com"
4. 点击"发送重置链接"
5. 收到重置邮件
6. 点击链接，设置新密码
7. 用新密码登录成功
```

## 填写说明

### 用户故事格式
```
作为 {角色}，我想要 {功能}，以便 {价值}。
```
- **角色**：谁会使用这个功能
- **功能**：他们想要什么功能
- **价值**：这个功能给他们带来什么价值

### 验收标准格式
```
给定 {前提}，当 {操作}，则 {结果}。
```
- **前提**：测试的前置条件
- **操作**：用户执行的操作
- **结果**：预期的结果

### 优先级定义
- **P0（必须）**：没有这个功能，系统无法使用
- **P1（重要）**：没有这个功能，用户体验很差
- **P2（可选）**：有这个功能更好，没有也不影响核心使用

---

## 注意事项

1. **每个需求必须有验收标准**：没有验收标准的需求无法验证
2. **验收标准要可测试**：避免"系统应该很快"这种模糊描述
3. **边界要明确**：明确做什么、不做什么，避免范围蔓延
4. **依赖要完整**：列出所有依赖，包括已完成和待开发的
5. **验收场景要具体**：具体的步骤和预期结果，便于验证
