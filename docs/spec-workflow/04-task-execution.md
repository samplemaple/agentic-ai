# 任务执行规范

> 按任务块分 session 执行，每个任务块完成后验证。
> 这是整个流程中耗时最长的阶段。

---

## 执行流程总览

```
读取执行计划 → 找到待执行任务块 → 读取任务详情 → 执行任务 → 验证 → 更新状态
                                                                      ↓
                                                              下一个任务块
```

---

## 步骤 1：读取执行计划

### 1.1 找到 tasks.md
读取当前 spec 的 tasks.md 文件。

### 1.2 找到执行计划
执行计划在 tasks.md 的顶部，格式如下：

```markdown
## 执行计划

| 任务块 | 包含任务 | 预估上下文 | 依赖 | 状态 |
|--------|---------|-----------|------|------|
| A | task1, task2, task3 | ~60k | 无 | ⬜ |
| B | task4, task5 | ~250k | A完成 | ⬜ |
| C | task6（测试） | ~40k | B完成 | ⬜ |
```

### 1.3 确认当前待执行任务块
找到状态为 `⬜` 的第一个任务块。

**检查依赖**：
- 如果依赖的任务块状态不是 `✅`，不能执行
- 等待依赖完成后再执行

---

## 步骤 2：读取任务详情

### 2.1 读取任务块说明
每个任务块有详细说明，格式如下：

```markdown
### 任务块 A：{标题}

**产出文件**：
- {文件1} — {描述}
- {文件2} — {描述}

**修改文件**：
- {文件1} — {改动摘要}
- {文件2} — {改动摘要}

**测试任务**：
- 测试文件：{路径}
  - 测试点1
  - 测试点2

**验证**：{编译/测试命令}
```

### 2.2 读取需要修改的现有文件
读取"修改文件"列表中的所有文件，了解当前代码状态。

### 2.3 读取 implementation-notes.md
读取实施注意事项，特别是：
- 关键设计决策
- 现有代码改造清单
- 高风险改造说明
- 测试注意事项

---

## 步骤 3：执行任务

### 3.1 按任务顺序执行
按任务块中的任务顺序执行：

1. 实现产出文件
2. 修改现有文件
3. 编写测试文件

### 3.2 实现产出文件
根据任务详情，创建新文件：

```markdown
### task1: {标题}
**产出**：{文件路径} — {描述}
**实现要点**：
- {要点1}
- {要点2}
```

**实现原则**：
- 先写接口/类型定义
- 再写核心逻辑
- 最后写辅助函数

### 3.3 修改现有文件
根据改造清单，修改现有文件：

```markdown
### 改造清单
| 文件 | 改动 |
|------|------|
| {文件1} | {改动} |
| {文件2} | {改动} |
```

**修改原则**：
- 最小改动原则（只改必须改的）
- 保持现有代码风格
- 不要重构不相关的代码

### 3.4 编写测试文件
根据测试任务，编写测试：

```markdown
### 测试任务
- 测试文件：{路径}
  - 测试点1：{描述}
  - 测试点2：{描述}
```

**测试原则**：
- 先写核心功能的测试
- 覆盖正常路径和边界情况
- 测试要独立，不依赖执行顺序

---

## 步骤 4：验证

### 4.1 编译验证
运行编译命令，确保无错误：

```bash
# 根据项目类型选择命令
# Python: python -m py_compile {file}
# Node.js: npm run build
# Go: go build ./...
# Rust: cargo build
# Java: mvn compile
```

**编译失败处理**：
1. 读取错误信息
2. 定位问题文件和行号
3. 修复错误
4. 重新编译
5. 重复直到编译通过

### 4.2 测试验证
运行测试命令，确保全部通过：

```bash
# 根据项目类型选择命令
# Python: pytest
# Node.js: npm test
# Go: go test ./...
# Rust: cargo test
# Java: mvn test
```

**测试失败处理**：
1. 读取失败的测试
2. 分析失败原因（代码 bug 还是测试问题）
3. 修复代码或测试
4. 重新测试
5. 重复直到全部通过

### 4.3 回归验证
确保已有测试没有被破坏：

```bash
# 运行完整测试套件
{测试命令}
```

**回归失败处理**：
1. 找到失败的已有测试
2. 分析是否由本次改动引起
3. 如果是，修复兼容性问题
4. 如果不是，记录并跳过

---

## 步骤 5：更新状态

### 5.1 更新任务块状态
在 tasks.md 中，将任务块状态从 `⬜` 改为 `✅`：

```markdown
| 任务块 | 包含任务 | 预估上下文 | 依赖 | 状态 |
|--------|---------|-----------|------|------|
| A | task1, task2, task3 | ~60k | 无 | ✅ |
| B | task4, task5 | ~250k | A完成 | ⬜ |
```

### 5.2 追加已完成摘要
在 tasks.md 的"已完成任务块摘要"部分追加：

```markdown
### 任务块 A 已完成（{日期}）

**产出文件**：
- {文件1} — {描述}
- {文件2} — {描述}

**修改文件**：
- {文件1} — {改动摘要}
- {文件2} — {改动摘要}

**验证结果**：
- 编译：✅ 通过
- 测试：✅ 通过（{N} 个用例）

**关键决策**：
- {决策1} — {理由}
- {决策2} — {理由}

**踩坑记录**：
- {问题} — {解决方案}
```

---

## 步骤 6：启动下一个任务块

### 6.1 检查是否还有未完成的任务块
在执行计划中查找状态为 `⬜` 的任务块。

### 6.2 如果有，开新 session
复制以下启动指令：

```
执行 {spec-name} 的下一个待执行任务块。

步骤：
1. 读 tasks.md 执行计划，确认当前待执行任务块
2. 读该任务块的详细说明（产出、修改、验证）
3. 读需要修改的现有文件
4. 实现产出 + 测试
5. 运行验证命令
6. 更新 tasks.md：任务块状态改为 ✅、追加已完成摘要
```

### 6.3 如果没有，进入验证阶段
所有任务块完成后，进入 [验证+归档阶段](01-spec-lifecycle.md#阶段-5验证--归档)。

## 中断恢复

如果 session 中断（上下文溢出、网络断开等），可以恢复：

### 恢复步骤

1. **读取 tasks.md**
   - 查看执行计划，找到最后一个 `✅` 的任务块
   - 确认下一个待执行任务块

2. **检查部分完成的任务块**
   - 如果任务块状态是 `⬜` 但有部分产出
   - 检查产出文件是否完整
   - 检查测试是否通过

3. **决定恢复策略**
   - 如果产出完整、测试通过 → 更新状态为 `✅`，继续下一个
   - 如果产出不完整 → 继续完成当前任务块
   - 如果测试失败 → 修复后继续

### 恢复指令

```
恢复 {spec-name} 的执行。

步骤：
1. 读 tasks.md 执行计划，查看当前状态
2. 找到最后一个 ✅ 的任务块
3. 检查下一个任务块是否有部分产出
4. 决定：继续当前任务块 / 开始新任务块
5. 执行并验证
6. 更新 tasks.md
```

## 上下文管理

### 上下文预算

| 内容 | 预估 token |
|------|-----------|
| 读取的源代码文件（每个） | 2k-10k |
| 编写的代码 + 编译修复 | 20k-50k |
| 测试代码 | 10k-20k |
| implementation-notes.md | 5k-10k |

**单任务块预算**：控制在 **100k-150k**，为意外情况留 50% 余量。

### 上下文优化技巧

1. **只读必要的文件**
   - 不要读取不相关的文件
   - 使用搜索定位具体代码，不要读取整个文件

2. **增量编写**
   - 先写核心逻辑，再写辅助代码
   - 先写简单测试，再写复杂测试

3. **及时验证**
   - 每完成一个任务就验证
   - 不要等到任务块完成才验证

4. **避免重复**
   - 不要重复读取已读过的文件
   - 不要重复执行已执行的命令

## 常见问题

### Q: 任务块执行时间太长怎么办？

**可能原因**：
- 任务块划分太大
- 遇到意外问题
- 上下文不足

**处理方式**：
1. 检查任务块是否可以进一步拆分
2. 如果遇到问题，先解决再继续
3. 如果上下文不足，开新 session 继续

### Q: 任务块依赖错误怎么办？

**可能原因**：
- 执行计划中的依赖关系错误
- 任务块划分不合理

**处理方式**：
1. 分析实际依赖关系
2. 调整执行计划
3. 重新划分任务块

### Q: 测试一直失败怎么办？

**可能原因**：
- 代码有 bug
- 测试本身有问题
- 环境问题

**处理方式**：
1. 先确认测试本身是否正确
2. 用调试工具定位代码问题
3. 如果是环境问题，检查依赖和配置

### Q: 发现设计缺陷怎么办？

**处理方式**：
1. 记录缺陷
2. 评估影响范围
3. 如果影响小，在 implementation-notes.md 中补充说明
4. 如果影响大，暂停执行，回到设计阶段修复

## 执行检查清单

每个任务块完成后，自查：

- [ ] 所有产出文件已创建？
- [ ] 所有修改文件已更新？
- [ ] 测试文件已编写？
- [ ] 编译通过？
- [ ] 测试通过？
- [ ] 已有测试无回归？
- [ ] tasks.md 状态已更新？
- [ ] 已完成摘要已追加？
