# 质量保障流程

> 三件套完成后、执行任务前，必须执行质量保障检查。
> 目的是提前发现问题，减少执行阶段的返工。

---

## 为什么需要质量保障

| 不做质量保障 | 做质量保障 |
|-------------|-----------|
| 执行时才发现设计缺陷 | 设计阶段就发现并修复 |
| 遗漏现有代码改造点 | 完整的改造清单 |
| 跨模块联动出错 | 提前识别联动约束 |
| 测试覆盖不全 | 明确的测试策略 |

---

## 质量保障三件套

### 1. 生成 implementation-notes.md

从三件套中提炼执行阶段需要的关键信息。

**标准章节**（按需选用，不必全部包含）：

#### 章节 1：关键设计决策及理由
```markdown
## 一、关键设计决策及理由

### 1.1 {决策名称}
- **做什么**：{具体实现}
- **为什么**：{理由}
- **不这样做**：{后果}

### 1.2 {决策名称}
...
```

**写法要点**：
- 每个决策一小节
- 说明"做什么 + 为什么 + 不这样做会怎样"
- 重点标注来自踩坑记录的决策

#### 章节 2：现有代码改造清单
```markdown
## 二、现有代码改造清单

| 文件 | 当前代码 | 改造后代码 | 改动原因 |
|------|---------|-----------|---------|
| {文件1} | {现状} | {目标} | {原因} |
| {文件2} | {现状} | {目标} | {原因} |
```

**写法要点**：
- 这是最易出错的部分，必须完整
- 用搜索验证所有调用点
- 新增文件不会错，改错现有文件会导致回归 bug

#### 章节 3：高风险改造
```markdown
## 三、高风险改造

### 3.1 {改造名称}
- **风险**：{什么可能出错}
- **验证方法**：{如何验证正确性}
- **回滚方案**：{出错如何恢复}
```

**写法要点**：
- 涉及核心组件改造时必须有
- 详细说明兼容性风险
- 提供验证方法和回滚方案

#### 章节 4：跨模块联动约束
```markdown
## 四、跨模块联动约束

### 本模块为其他模块预留
- {字段/接口} — 供 {模块X} 使用

### 其他模块给本模块的待办
- {待办} — 来自 {模块X}

### 预留但不实现的功能
- {功能} — fallback 行为：{说明}
```

**写法要点**：
- 明确标注"本模块做"和"其他模块做"的边界
- 预留功能必须说明 fallback 行为

#### 章节 5：测试注意事项
```markdown
## 五、测试注意事项

### 单元测试
- {注意事项1}
- {注意事项2}

### 集成测试
- {注意事项1}
- {注意事项2}

### 端到端测试
- {注意事项1}
```

**写法要点**：
- SO 创建方式、Mock 策略等
- 测试环境的特殊处理
- 已知的测试限制

#### 章节 6：文件组织
```markdown
## 六、文件组织

### 新增文件
| 目录 | 文件名 | 说明 |
|------|--------|------|
| {目录} | {文件} | {描述} |

### 修改文件
| 文件 | 改动摘要 |
|------|---------|
| {文件} | {描述} |
```

### 2. 检查清单

AI 必须逐条检查，全部通过后才能进入执行阶段。

#### 需求检查
```markdown
### 需求检查
- [ ] 每个需求有用户故事和验收标准
- [ ] 术语表覆盖所有新增概念
- [ ] 边界明确（哪些不做）
- [ ] 优先级划分清晰（P0/P1/P2）
```

#### 设计检查
```markdown
### 设计检查
- [ ] 架构图清晰展示组件关系
- [ ] 接口定义完整（参数、返回值、异常）
- [ ] 数据模型覆盖所有场景
- [ ] 错误处理策略明确
- [ ] 测试策略区分单元/集成/端到端
```

#### 任务检查
```markdown
### 任务检查
- [ ] 任务按依赖顺序排列
- [ ] 每个任务有明确的产出和验证方式
- [ ] 任务块划分合理（上下文预算 100k-150k）
- [ ] 高风险改造独立成批

### 高风险改造检查
以下改造是否已识别并独立成批：
- [ ] 构造函数签名变更
- [ ] 继承链修改
- [ ] 接口签名变更
- [ ] 枚举值变更（插入/删除）
- [ ] 数据库 schema 变更
- [ ] 核心组件重构
- [ ] API 接口变更
- [ ] 依赖注入变更
```

#### implementation-notes 检查
```markdown
### implementation-notes 检查
- [ ] 关键设计决策有理由说明
- [ ] 现有代码改造清单完整（用搜索验证）
- [ ] 高风险改造有验证方案
- [ ] 预留功能说明了 fallback 行为
- [ ] 文件组织列出了所有新增和修改的文件
```

### 3. 现有代码审查

如果 spec 涉及修改现有代码，必须执行现有代码审查。

#### 3.1 全局 API 变更审查
当引入新的 API 变更时：

1. 搜索被修改的方法/类的所有调用点
2. 逐个判断：该调用点是否需要改造
3. 列入改造清单

```
搜索示例：
- 方法名变更：搜索旧方法名的所有调用
- 参数变更：搜索方法名，检查参数使用
- 返回值变更：搜索方法名，检查返回值使用
```

#### 3.2 继承链变更审查
当修改类的继承关系时：

1. 搜索所有 `new ClassName()` 或 `CreateInstance<ClassName>()`
2. 搜索所有 `FindObjectOfType<ClassName>()` 或类似查找
3. 搜索所有 `is ClassName` 或 `as ClassName` 类型检查
4. 判断是否需要改造

#### 3.3 枚举扩展审查
当扩展现有枚举时：

1. 搜索所有 switch/if-else 分支
2. 判断是否需要覆盖新值
3. 检查序列化兼容性

#### 3.4 接口变更审查
当修改接口时：

1. 搜索所有接口实现类
2. 搜索所有接口调用点
3. 判断是否需要改造

---

## 质量检查流程

### 步骤 1：生成 implementation-notes.md
AI 根据三件套生成 implementation-notes.md。

### 步骤 2：执行检查清单
AI 逐条检查，记录结果。

### 步骤 3：执行现有代码审查
如果涉及修改现有代码，执行审查。

### 步骤 4：修复问题
如果检查发现问题：
- 需求问题 → 更新 requirements.md
- 设计问题 → 更新 design.md
- 任务问题 → 更新 tasks.md

### 步骤 5：确认通过
所有检查通过后，进入执行阶段。

## 常见问题

### Q: implementation-notes.md 和 design.md 有什么区别？

**design.md**：设计层面的决策（架构、接口、数据模型）
**implementation-notes.md**：执行层面的注意事项（改造清单、踩坑、测试技巧）

design.md 是"做什么"，implementation-notes.md 是"怎么做"。

### Q: 所有 spec 都需要 implementation-notes.md 吗？

**建议**：
- 简单 spec（任务 < 5，不涉及现有代码修改）：可选
- 中等 spec（任务 5-15，涉及少量现有代码修改）：推荐
- 复杂 spec（任务 > 15，涉及大量现有代码修改）：必须

### Q: 检查清单没通过怎么办？

**处理方式**：
1. 定位问题所在（需求/设计/任务）
2. 修复对应文档
3. 重新检查
4. 全部通过后才能进入执行阶段

### Q: 现有代码审查发现遗漏怎么办？

**处理方式**：
1. 补充到 implementation-notes.md 的改造清单
2. 如果影响设计，更新 design.md
3. 如果影响任务，更新 tasks.md
4. 重新检查清单
