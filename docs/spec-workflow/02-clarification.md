# 提问澄清规范

> 在生成三件套之前，AI 必须先问澄清问题，确保理解正确。
> 这是减少返工、提高设计质量的关键步骤。

---

## 为什么需要提问

| 不提问 | 提问后 |
|--------|--------|
| AI 猜测用户意图 | 用户明确表达意图 |
| 生成后发现方向错 | 方向确认后再生成 |
| 反复修改三件套 | 一次生成到位 |
| 浪费上下文 | 高效利用上下文 |

---

## 标准问题清单

### 核心问题（必须问，5-8 个）

#### 1. 核心场景
```
这个功能的主要使用场景是什么？
请描述 1-2 个最常见的用户故事。
```

**目的**：理解功能的核心价值，避免过度设计。

#### 2. 边界条件
```
哪些情况应该处理？哪些不应该处理？
有没有明确的边界或限制？
```

**目的**：明确 scope，避免范围蔓延。

#### 3. 优先级
```
如果有时间压力，哪些功能可以砍掉？
哪些是"必须有"，哪些是"最好有"？
```

**目的**：帮助划分 MVP 和后续迭代。

#### 4. 依赖
```
这个功能依赖哪些现有系统？
有没有需要先完成的前置工作？
```

**目的**：识别依赖关系，避免阻塞。

#### 5. 验收标准
```
怎样算"完成"？
有没有具体的验收标准或测试场景？
```

**目的**：明确"完成"的定义，避免歧义。

#### 6. 技术约束
```
有没有必须遵循的技术限制？
比如：性能要求、兼容性要求、安全要求？
```

**目的**：识别硬约束，避免后期返工。

#### 7. 扩展性
```
未来可能如何扩展？
有没有预留接口的需求？
```

**目的**：设计时考虑扩展性，避免架构僵化。

#### 8. 风险
```
你觉得哪些地方可能有坑？
有没有之前踩过的坑需要注意？
```

**目的**：提前识别风险，制定应对策略。

---

### 可选问题（根据 spec 类型选择）

#### 涉及 UI 时
```
交互风格偏好？（简洁/丰富、桌面/移动）
视觉风格？（颜色、字体、动效）
有没有参考的产品或设计？
```

#### 涉及性能时
```
性能目标是什么？（响应时间、吞吐量、内存）
已知的性能瓶颈在哪？
有没有性能测试的要求？
```

#### 涉及安全时
```
安全要求是什么？（认证、授权、加密）
威胁模型是什么？（哪些攻击向量）
有没有合规要求？（GDPR、SOC2）
```

#### 涉及集成时
```
第三方 API 的文档在哪？
数据格式是什么？（JSON、XML、Protobuf）
有没有 rate limit 或配额限制？
```

#### 涉及数据时
```
数据量级是多少？（记录数、并发量）
数据保留策略？（多久、多大）
有没有数据迁移需求？
```

#### 涉及状态管理时
```
状态流转是怎样的？
有没有需要持久化的状态？
并发场景下的状态一致性要求？
```

## 提问流程

### 步骤 1：AI 分析 spec 范围
AI 根据用户的功能描述，识别：
- 涉及的技术领域
- 可能的复杂度
- 需要澄清的关键点

### 步骤 2：AI 生成问题清单
AI 生成 5-8 个问题，格式如下：

```markdown
## 澄清问题

基于你描述的功能，我有以下问题需要确认：

### 1. 核心场景
{问题}

### 2. 边界条件
{问题}

### 3. 优先级
{问题}

### 4. 依赖
{问题}

### 5. 验收标准
{问题}

### 6. 技术约束
{问题}

### 7. 扩展性
{问题}

### 8. 风险
{问题}

请逐一回答，我会根据你的回答生成三件套。
```

### 步骤 3：用户回答
用户逐一回答问题。可以：
- 详细回答（推荐）
- 简短回答（AI 会追问模糊的地方）
- 跳过（表示由 AI 决定）

### 步骤 4：AI 确认理解
AI 总结用户回答，确认理解正确：

```markdown
## 我的理解

基于你的回答，我理解这个 spec 是：

**核心功能**：{一句话总结}

**范围**：
- ✅ 做：{列表}
- ❌ 不做：{列表}

**关键约束**：
- {约束1}
- {约束2}

**优先级**：
- P0（必须）：{列表}
- P1（重要）：{列表}
- P2（可选）：{列表}

**风险点**：
- {风险1} — 应对：{策略}
- {风险2} — 应对：{策略}

请确认我的理解是否正确，或补充/修正。
```

### 步骤 5：用户确认
用户确认或修正。确认后进入三件套生成。

---

## 复杂度评估

基于用户回答，AI 自动评估 spec 复杂度：

### 评估维度

| 维度 | 低 | 中 | 高 |
|------|----|----|-----|
| 任务数量 | < 10 | 10-20 | > 20 |
| 涉及文件 | < 5 | 5-15 | > 15 |
| 依赖系统 | 0-1 | 2-3 | > 3 |
| 风险等级 | 低 | 中 | 高 |

### 评估结果

```markdown
## 复杂度评估

基于你的回答，我评估这个 spec：

| 维度 | 评估 |
|------|------|
| 任务数量 | 约 {N} 个 |
| 预估上下文 | ~{X}k token |
| 涉及现有文件 | {N} 个 |
| 风险等级 | {低/中/高} |

**建议**：{不分批 / 分 2-3 个任务块 / 分 4+ 个任务块}

**理由**：
- {理由1}
- {理由2}
```

### 分批策略

| 评估结果 | 建议 |
|---------|------|
| 任务 < 10，上下文 < 150k | 不分批，一个 session 完成 |
| 任务 10-20，上下文 150k-300k | 可选分批，根据风险决定 |
| 任务 > 20，上下文 > 300k | 必须分批 |

**高风险改造必须独立成批**（无论任务数量）：
- 构造函数签名变更
- 继承链修改
- 接口签名变更
- 数据库 schema 变更
- 核心组件重构
| 涉及高风险改造 | 必须分批，高风险任务独立成块 |

## 常见问题处理

### 用户回答太简短
AI 追问：
```
你提到"支持多种登录方式"，能具体说说是哪些方式吗？
比如：邮箱+密码、手机号+验证码、第三方 OAuth？
```

### 用户回答有矛盾
AI 指出并确认：
```
你提到"性能要求很高"，但又说"先快速实现"。
这两个目标可能冲突，你更优先哪个？
```

### 用户不确定
AI 提供建议：
```
你不确定用哪种方案，我建议用方案 A，因为：
- {理由1}
- {理由2}

你同意吗？或者你有其他考虑？
```

### 用户要求太宽泛
AI 缩小范围：
```
你提到"做一个完整的用户系统"，这个范围很大。
建议先聚焦核心功能：
- 第一期：注册、登录、个人资料
- 第二期：权限管理、角色系统
- 第三期：社交功能

你同意这个分期吗？
```

## 提问质量检查清单

AI 生成问题后，自查：

- [ ] 是否覆盖了所有核心维度（场景、边界、优先级、依赖、验收、约束、扩展、风险）？
- [ ] 问题是否具体（避免"你觉得呢"这种泛泛的问题）？
- [ ] 问题数量是否合适（5-8 个，不要太多让用户疲劳）？
- [ ] 是否根据 spec 类型添加了可选问题？
- [ ] 问题顺序是否合理（从核心到细节）？

## 示例

### 示例 1：用户认证系统

**用户描述**：
```
我要做一个用户认证系统，支持注册、登录、密码重置。
```

**AI 生成的问题**：

```markdown
## 澄清问题

### 1. 核心场景
用户认证的主要使用场景是什么？
是 Web 应用、移动应用、还是 API 服务？

### 2. 边界条件
- 需要支持第三方登录吗？（Google、GitHub、微信）
- 需要支持手机号登录吗？
- 需要支持多因素认证（MFA）吗？

### 3. 优先级
如果有时间压力，哪些可以砍掉？
- 必须有：{猜测：注册、登录}
- 重要：{猜测：密码重置}
- 可选：{猜测：第三方登录、MFA}

### 4. 依赖
- 用什么数据库？（PostgreSQL、MongoDB、MySQL）
- 用什么框架？（Express、FastAPI、Spring Boot）
- 有没有现有的用户表或认证模块？

### 5. 验收标准
怎样算"完成"？
- 注册：邮箱验证？用户名唯一性？
- 登录：JWT token？Session？
- 密码重置：邮件发送？链接有效期？

### 6. 技术约束
- 安全要求？（密码哈希算法、token 过期时间）
- 性能要求？（并发登录数）
- 合规要求？（GDPR、数据保留）

### 7. 扩展性
未来可能扩展哪些功能？
- 单点登录（SSO）？
- OAuth2.0？
- 用户画像？

### 8. 风险
你觉得哪些地方可能有坑？
- 安全性？（SQL 注入、XSS、CSRF）
- 并发？（竞态条件）
- 数据一致性？（分布式场景）
```

### 示例 2：购物车系统

**用户描述**：
```
我要做一个购物车系统，支持添加商品、修改数量、结算。
```

**AI 生成的问题**：

```markdown
## 澄清问题

### 1. 核心场景
购物车的使用场景是什么？
- 电商网站？
- 移动 App？
- 小程序？

### 2. 边界条件
- 需要支持游客购物车吗？（未登录用户）
- 需要支持收藏夹吗？
- 需要支持商品规格选择吗？（颜色、尺寸）

### 3. 优先级
- 必须有：{猜测：添加、删除、修改数量}
- 重要：{猜测：结算、库存检查}
- 可选：{猜测：收藏、分享}

### 4. 依赖
- 商品系统在哪？（已有还是需要新建）
- 库存系统在哪？
- 支付系统在哪？

### 5. 验收标准
- 添加商品：库存不足时怎么处理？
- 结算：支持哪些支付方式？
- 并发：多人同时购买同一商品？

### 6. 技术约束
- 性能要求？（购物车加载时间）
- 数据持久化？（购物车数据保留多久）
- 移动端适配？

### 7. 扩展性
- 优惠券系统？
- 推荐系统？
- 跨设备同步？

### 8. 风险
- 库存超卖？
- 价格变动？
- 并发竞态？
```
